# Cover Letter Pitch

JD를 입력받으면 서류 합격 확률을 최대화하는 **맞춤형 자기소개서**를 생성한다.
사실을 나열하는 것이 아니라, 후보자의 판단 흐름과 임팩트를 JD의 Pain Point에 맞춰 어필하는 것이 목표다.

> **언어**: 사용자의 대화 언어를 따른다. 한국어 요청이면 한국어로, 영어 요청이면 영어로 작성한다.
> 아래 절차·예시는 한국어 채용 맥락 기준이며, 영어 커버레터에도 동일한 구조를 적용한다.

---

## 합격 전략 개요

자기소개서는 결국 사람을 설득하는 글이다.

```
JD 입력
  └─► JD 해부 → 후보자 경험 매핑 → 어필 문장 재작성
        └─► 사람 필터 — 리크루터(30초) → 팀장/시니어(5분)
              └─► 서류 합격
```

리크루터와 팀장은 "이 사람을 뽑고 싶다"는 판단을 원한다.
**사실 나열은 정보는 전달하지만, "왜 이 사람이어야 하는가"에 답하지 못해 탈락한다.**
자기소개서는 "무엇을 했는가"가 아니라 "왜 이 회사에 이 사람이어야 하는가"를 증명해야 한다.

---

## Step 0 — 입력 확보 (작성 시작 전 필수)

이 스킬은 후보자 정보를 내부에 저장하지 않는다. **모든 경험·수치는 사용자가 제공한 소스에서만 가져온다.**

**필수 입력 — 없으면 작성하지 말고 먼저 요청한다**

1. **JD 전문** — 채용공고 텍스트 또는 파일 경로/URL
2. **이력서** — PDF·docx·md 등 파일 경로 또는 붙여넣은 텍스트 (대부분의 사용자가 가진 표준 입력)
3. **목표 언어** — 한국어 / 영어
4. **포지션 레벨** — 주니어 / 미드 / 시니어 / 리드

**선택 입력 — 있으면 근거가 풍부해진다**

- **포트폴리오** — 이력서를 보완하는 자료. **PDF 파일** / 웹 링크 / 이미지 / 붙여넣은 텍스트 모두 가능하다. 프로젝트 상세·수치·맥락이 담겨 있어 인터뷰(Step 2.5)에서 물어볼 항목을 줄여준다.

**있으면 받고, 없으면 작성 전에 묻는다 (품질·합격을 좌우)**

- **지원서 문항**: 자유 양식인가 / 정해진 문항형인가 (문항형이면 문항별로 작성)
- **글자 수·분량 제한**: 문항별 또는 전체 (있으면 반드시 준수, 초과 시 즉시 탈락 가능)
- **회사·제품 정보**: 회사명, 제품, 최근 방향 (3단락 기여 시나리오 구체화에 사용)
- **이력서 동봉 여부**: 동봉되면 자소서에서 이력서 내용을 반복하지 않고 보완·해석에 집중

> **입력 누락 처리**: 위 항목이 비어 있으면 추정하지 말고 사용자에게 먼저 묻는다. 특히 **문항형 여부와 글자 수 제한**은 형식 미준수로 즉시 탈락할 수 있으므로 반드시 확인한다.

## Step 1 — JD 해부 (JD Decomposition)

JD를 다음 5개 레이어로 분해한다.

| 레이어 | 추출 대상 | 목적 |
|--------|----------|------|
| **Required Skills** | "필수", "must have", 반복 등장 기술 | 반드시 증명해야 할 핵심 역량 |
| **Preferred Skills** | "우대", "nice to have" | 차별화 포인트 |
| **Seniority Signal** | 연차 요구, 표현 톤(lead/contribute/support) | 경험 서술 깊이 조정 |
| **Culture Signal** | 회사 소개, 팀 설명의 형용사 | 자기소개서 톤 정렬 |
| **Pain Point (가설)** | JD가 이 포지션을 열게 된 이유에 대한 가설 | 자기소개서가 정조준할 과제 |

> **Pain Point는 '사실'이 아니라 '가설'이다.** JD만으로 회사 내부 사정을 단정할 수 없다. 역추론은 **어디를 겨냥할지 정하는 내부 도구**일 뿐, 자기소개서 본문에서 회사의 약점을 단정하지 않는다.
> - 내부 분석에서는 자유롭게 가설을 세운다.
>   예) "빠르게 성장하는 서비스" → 스케일링 부담이 큰 시점일 가능성.
>   예) "AI 기능 도입 확대" → AI를 주도할 인력이 더 필요한 국면으로 보임.
>   예) "주도적 참여"를 자격 요건에 명시 → 스펙 수동 구현을 넘는 사람을 찾고 있다고 해석.
> - **본문 표현 규칙**: 단정형("귀사는 ~가 부족합니다") 금지. 가설형("이 포지션에서 중요해 보이는 과제는 ~일 것 같습니다", "~한 국면이라면")으로만 언급하고, 곧바로 "그 지점에서 제가 이렇게 기여할 수 있습니다"로 연결한다. 진단이 아니라 기여 제안의 도입부로만 쓴다.

## Step 2 — 후보자 경험 매핑 + 리스크 분석

후보자의 경험을 JD 레이어에 매핑하고, 공백을 솔직하게 파악한다.

**Step 2.0 — 경험 전수 나열 (Inventory, 필터링 전 필수)**

선택하기 전에 먼저 **전부 펼친다.** 큰 프로젝트뿐 아니라 작은 기능·사이드·일회성 작업까지 후보자가 제공한 소스에서 빠짐없이 끌어모은다.
- 누락은 "선택"이 아니라 "애초에 후보에 못 올라간 것"에서 발생한다. 보이지 않으면 고를 수 없다.
- 특히 **프로필의 주력과 다른 결의 경험**을 의식적으로 찾는다. 서버 중심 이력이라면 프론트/풀스택 경험, 백엔드 중심이라면 운영·기획 경험.
- 각 경험에 "이게 증명하는 능력(역량 차원)"을 한 줄로 태깅한다. 예) 사내 도구를 혼자 화면부터 API까지 만든 경험 → *프론트+백엔드 end-to-end 구현 능력*.

> **실패 사례 (반드시 회피)**: 풀스택 JD인데 후보 이력이 한쪽(예: 서버)에 쏠려 있던 상황에서, 작게라도 화면까지 직접 만든 경험을 끄집어내지 못했다. 그게 풀스택을 증명하는 **가장 결정적인 단 하나의 증거**였는데, 주력 경험에 밀려 후보 목록에조차 오르지 못했다. 전수 나열을 건너뛰면 이 누락이 반복된다. (직군 불문: 데이터 직군이라면 분석→대시보드까지 직접 한 경험, 백엔드라면 운영·장애 대응 경험이 그런 '유일한 증거'일 수 있다.)

```
JD Required Skills
  └─► 직접 매칭 → 반드시 자소서에 녹임, 수치 보강
  └─► 매칭 없음 → 가장 근접한 경험 + 전이 가능성 명시, 선제 언급

JD Pain Point
  └─► 유사 문제 해결 경험 → 자소서 도입부·핵심 단락에 배치
```

**매핑 우선순위 규칙**

선택은 감이 아니라 아래 3개 신호로 가른다 (점수 합산이 아니라 우선순위 정렬 — 거짓 정밀도 경계).
- **직접성**: JD Required/Pain Point를 얼마나 직접 증명하는가 (직접 매칭 > 전이 가능)
- **증거 강도**: 수치·규모가 실재하는가 (정량 > 정성)
- **희소성**: 그 역량 차원을 증명하는 유일한 증거인가 (유일 증거는 가산점)

1. 위 신호로 정렬해 Pain Point 해결에 가장 직접적인 경험 최대 5개만 선택 — 많을수록 희석된다
2. **결정적 증거 보호 (개수 제한보다 우선)**: 어떤 JD Required 차원을 증명하는 경험이 **단 하나뿐**이라면, 최대 5개 제한과 무관하게 반드시 포함한다. "희석 방지"가 "유일 증거 누락"의 핑계가 되어선 안 된다.
   - 체크: JD Required/Pain Point 차원별로 매핑된 증거 개수를 센다 → 증거가 1개인 차원이 있으면 그 경험은 강제 채택.
   - 프로필 쏠림 경고: 선택한 최대 5개가 모두 같은 결(예: 전부 서버)이라면, JD가 요구하는 **다른 차원의 결정적 증거를 빠뜨렸을 가능성**을 의심하고 Step 2.0 인벤토리를 재확인한다.
3. 경험이 오래됐으면 최신 경험으로 대체하거나 현재도 유효함을 보강
4. Co-founder·창업자 타이틀은 반드시 **실제 개발(또는 실무) 비중**을 명시 — 타이틀만으로는 경영·기획 역할로 오해받는다. 예: "(AI·백엔드 개발 전담)". 실제로 맡은 범위만 쓰고, 하지 않은 역할을 적지 않는다.

**커버 불가능한 공백 처리 원칙**
- **선제 언급은 JD 핵심 Required와 직접 충돌하는 공백에만 적용한다.** 사소한 Preferred 공백이나 누구나 갖는 약점은 먼저 꺼내지 않는다 — 불필요한 약점 강조는 손해다.
- 핵심 공백은 숨기지 않는다. 면접관은 어차피 발견한다.
- 핵심 공백 선제 언급 구조 → "X는 경험이 없지만, Y로 즉시 기여 가능하며 X는 온보딩 중 학습 의지 있음"
- 핵심 약점을 먼저 꺼내는 사람이 숨기는 사람보다 신뢰도가 높다

## Step 2.5 — 근거 보강 인터뷰 (이력서가 빈약할 때 필수)

이력서는 보통 "FastAPI로 주문 API 개발" 같은 불릿만 있고, 이 스킬의 핵심인 **판단 이유·수치·임팩트**가 빠져 있다. 없는 근거를 지어내는 대신, **작성 전에 사용자에게 물어 채운다.**

1. Step 2에서 선택한 결정적 경험(최대 5개) 각각에 대해 세 요소가 소스(이력서·포트폴리오)에 있는지 점검한다:
   - ① 그 기술/구조/판단을 **선택한 이유** (어떤 문제 때문에?)
   - ② **수치·규모** (응답시간, 처리량, 사용자 수, 절감폭, 기간 등)
   - ③ **임팩트(결과)** — 그래서 무엇이 달라졌는가
2. 포트폴리오가 있으면 **먼저 참고해 자동으로 채울 수 있는 항목은 질문에서 뺀다.**
3. 그래도 빠진 항목만 **한 번에 모아** 사용자에게 묻는다 (작성 전 1회, 결정적 경험에 한정 — 질문 폭탄 금지).
   - 예: "이력서엔 '주문 API 개발'만 있는데, 어떤 문제 때문에 그 구조를 택했나요? 응답시간·처리량 같은 수치가 있나요?"
4. 사용자 답변은 **정당한 근거**다. 답변 내용을 본문 근거로 사용하고, Step 5 검증표의 출처로 기록한다.
5. 사용자가 답을 못 주는 항목은 **추정하지 말고** 정성적 임팩트로 대체한다 (없는 수치 날조 금지 — Step 3·원칙 준수).

## Step 3 — 경험 문장 재작성: 사실이 아닌 어필

**핵심 원칙: 무엇을 했는지가 아니라, 왜 그 판단을 했고 결과가 무엇이었는지를 쓴다.**

직군과 무관하게 같은 원칙이 적용된다. 아래는 직군별 예시다.

```
[프론트엔드]
나쁜 예 — React로 상품 목록 페이지 구현
좋은 예 — 초기 로딩 4초로 이탈이 컸던 상품 목록을, 병목을 프로파일링해
         코드 스플리팅·이미지 지연 로딩·가상 스크롤을 적용. 로딩을 1.2초로 줄여
         이탈률을 눈에 띄게 낮췄다.
```

```
[백엔드]
나쁜 예 — 주문 API 설계·구현
좋은 예 — 피크 시간 타임아웃이 잦던 주문 API를, "동기 처리 구조가 병목"이라 판단해
         결제·재고 차감을 비동기 큐로 분리. p99 응답을 절반으로 줄이고
         피크 트래픽에서도 무중단 처리.
```

```
[데이터 엔지니어]
나쁜 예 — Airflow로 일배치 파이프라인 구축
좋은 예 — 매일 새벽 실패하던 배치를, 의존성·재처리 비용을 분석해
         증분 적재 + 멱등 설계로 재설계. 실패 시 전체 재처리 대신 구간 재처리만으로
         복구 시간을 수 시간에서 수 분으로 단축.
```

> 위 예시는 형식을 보여주기 위한 것이다. 수치도 예시일 뿐이다. **실제 본문에는 후보자 소스에 실재하는 경험·수치만** 사용한다.

**재작성 체크리스트**
- [ ] 이 문장은 "내가 했다"인가, 아니면 "왜 이 판단을 했고 결과가 이것이다"인가
- [ ] 기술/판단 선택 이유가 담겨 있는가 (단순 나열 금지)
- [ ] 수치 또는 규모가 포함되어 있는가
- [ ] **모든 수치가 후보자 소스(이력서·포트폴리오·인터뷰 답변)에 실재하는가** — 없으면 추정하지 말고 정성적 임팩트로 대체한다
- [ ] 수동태 금지 ("개발되었다" → "개발했다")
- [ ] JD 키워드가 자연스럽게 삽입되어 있는가

## Step 4 — 자기소개서 작성

자기소개서는 경험의 요약이 아니다. "왜 이 회사에 이 사람이어야 하는가"를 증명하는 글이다.

**구조 (4단락 — 자유 양식 기준. 문항형이면 문항별로 답하되 같은 원칙 적용)**

**1단락 — 나는 이런 사람이다 (도입부 선언)**
- 직함이나 스펙 나열 금지. 일하는 방식과 태도를 한 문장으로 선언.
- 예: "저는 스펙을 구현하는 사람이 아니라, 문제를 찾아서 푸는 사람입니다."

**2단락 — JD에서 이 문장이 눈에 들어왔다 (공명 증명)**
- JD의 자격 요건 또는 팀 설명에서 핵심 문장을 직접 인용.
- "이 문장이 자격 요건에 있다는 것은 ~라는 뜻입니다"로 해석.
- 그 요건을 내가 충족한다는 구체적 사례(수치 포함)로 즉시 증명.
- 예: "'주어진 사양대로만 개발하는 것을 넘어 주도적으로 참여' — 저는 팀이 반복하던 수작업을 발견해 자발적으로 자동화 도구로 만들었습니다."

**3단락 — 이 회사에서 하고 싶은 것 + 내가 줄 수 있는 가치 (지원 동기 × 기여 제안)**
- **이 단락은 다음 두 질문에 정면으로 답해야 한다 (모든 자기소개서의 필수 항목)**:
  - "당신이 우리 회사에 오면 어떤 가치를 제공해줄 수 있습니까?"
  - "우리가 당신을 뽑으면 어떤 이점이 생깁니까?"
- 이 질문은 회사 입장의 ROI 질문이다. "내가 무엇을 배우고 싶다"가 아니라 **"회사가 무엇을 얻는다"**로 답한다. 주어를 '나의 성장'이 아니라 '회사가 얻는 것'에 둔다.
- "귀사의 성장에 기여하고 싶습니다" 금지. 막연한 의지 표명이 아니라 **구체적인 기여 시나리오**를 제시한다.
- 회사의 구체적인 제품·기술 방향을 언급하고, 내 경험이 어떻게 연결되는지 구체적으로 서술.
- "이 포지션에서 중요해 보이는 과제(Pain Point 가설)"를 가설형으로 언급하고, **내가 합류하면 그 과제를 어떤 방식으로 풀 수 있는지**를 1인칭으로 제안한다. (회사의 결함을 단정하지 않는다 — Step 1의 본문 표현 규칙 준수)
- 가치 어필은 다음 3개 축으로 구성한다:
  - **즉시 기여 (Day 1)**: 온보딩 없이 바로 투입 가능한 영역. JD Required와 직접 매칭되는 경험 기반. "OO는 바로 맡아 진행할 수 있습니다."
  - **역할 정의 (어떤 포지션으로 설지)**: 단순 실행자가 아니라 어떤 역할을 자처할지 선언. 예) "기능 구현뿐 아니라 코드 품질·성능 구조를 설계하는 역할을 맡고 싶습니다." (직군에 맞게: 프론트=UX·렌더링 성능 구조 / 백엔드=확장성·장애 대응 구조 / 데이터=파이프라인 신뢰성·비용 구조)
  - **확장 기여 (6~12개월)**: 시간이 지나며 만들어낼 임팩트. 0→1 경험, 운영 도구화, 팀 레버리지 등.
- 어필은 반드시 **과거 실증(후보자 소스의 경험·수치)에 근거**해야 한다. "할 수 있습니다"는 "전에 OO를 해서 OO를 만들었기 때문에 여기서도 할 수 있습니다" 구조로.
- 금지: 회사가 시킬 일을 그대로 받아 적는 식("귀사가 필요로 하는 OO를 하겠습니다"). 내가 정의하는 역할을 제안하는 톤이어야 한다.

**4단락 — 약점 선제 언급 (조건부, 신뢰 구축)**
- **조건**: JD 핵심 Required와 직접 충돌하는 공백일 때만 짧게 선제 언급한다. 사소한 Preferred 공백이나 누구나 갖는 약점은 언급하지 않는다 — 불필요한 약점 강조는 오히려 손해다.
- 충돌하는 공백이 없으면 이 단락은 생략하고 3단락(기여 가치)으로 마무리한다.
- 구조(언급할 경우): "X는 경험이 없습니다. 대신 Y로 즉시 기여 가능하며, X는 온보딩 중 병행 학습할 의지가 있습니다."
- 핵심 공백을 숨기면 면접관이 어차피 발견한다. 먼저 말하는 사람이 더 신뢰받는다 — 단, "먼저 말한다"는 핵심 공백에만 적용한다.

### 포지션 레벨별 톤 조절

같은 경험도 레벨에 따라 어필 축을 바꾼다. 레벨에 맞지 않으면 주니어는 건방져 보이고, 시니어는 작아 보인다.

| 레벨 | 3단락 무게중심 | 톤 |
|------|---------------|-----|
| **주니어** | 즉시 기여 가능 영역 + 학습 속도 | 겸손하되 구체적. 역할 정의·시스템 개선 주장은 절제 |
| **미드** | 문제 해결 방식(판단 흐름) + 즉시 기여 | 자기 판단 근거를 드러내되 단정은 자제 |
| **시니어/리드** | 역할 정의 + 시스템·구조 개선 + 팀 레버리지 | 어떤 역할로 설지 주도적으로 선언 |

## Step 5 — 최종 검토 (본문 확정 전 내부 검증)

> 순서가 곧 작업 순서다. 본문을 먼저 완성해 내보낸 뒤 검증하는 것이 아니라, **검증표·형식 점검을 통과시킨 내용만 제출용 본문으로 확정**한다.

**주장-근거 검증표 (제출 전 필수, 내부 도구)**

본문을 내보내기 전에, 자기소개서의 모든 핵심 주장·수치를 표로 뽑아 근거를 대조한다. 근거 없는 문장은 고치거나 삭제한다.

| 본문 주장/수치 | 근거 출처 (이력서 / 포트폴리오 / 인터뷰 답변) | 판정 |
|---------------|---------------------------------------------|------|
| 예) "응답 지연 절반으로 단축" | 이력서 + 인터뷰 답변 | ✅ 실재 |
| 예) "비용 30% 절감" | (출처 못 찾음) | ❌ → 정성 표현으로 교체 또는 삭제 |

- 판정이 ❌인 항목은 **본문에 남기지 않는다.** 추정으로 메우지 않는다.
- 회사 사정에 대한 단정(Pain Point)이 본문에 단정형으로 남아 있지 않은지 함께 확인한다 → 가설형이어야 통과.

**형식 준수 점검 (문항형·글자수 제한이 있을 때)**
- [ ] 정해진 문항이 있으면 문항별로 답했는가 (한 덩어리로 뭉뚱그리지 않았는가)
- [ ] 글자 수·분량 제한을 초과하지 않았는가 (초과 시 즉시 탈락 가능)
- [ ] 이력서가 동봉되는 경우, 이력서 내용을 그대로 반복하지 않고 보완·해석에 집중했는가

**키워드 정렬**
- JD의 표현을 그대로 사용 (JD가 "k8s"면 "Kubernetes" 아닌 "k8s"로, "TS"면 "TypeScript" 아닌 "TS"로)
- Required Skills 키워드가 자기소개서 본문에 자연스럽게 등장하는가
- 키워드 스터핑 금지 — 문장 안에 자연스럽게 삽입

**30초 리크루터 체크리스트**
- [ ] 도입부에서 이 사람이 어떤 사람인지 즉시 파악 가능한가
- [ ] 핵심 경험과 임팩트가 한눈에 들어오는가

**5분 팀장 체크리스트**
- [ ] 경험 서술에 기술 판단 이유가 담겨 있는가 (단순 나열 금지)
- [ ] 규모와 임팩트를 수치로 확인 가능한가
- [ ] Co-founder·창업자 타이틀이 있다면, 개발(실무) 비중이 명시되어 있는가
- [ ] 이 사람이 합류하면 우리 Pain Point를 해결할 수 있겠다는 판단이 서는가

**결정적 증거 누락 점검 (제출 전 마지막 게이트)**
- [ ] JD Required 차원마다 그것을 증명하는 경험이 최소 1개씩 본문에 들어가 있는가
- [ ] 증거가 단 하나뿐인 차원의 경험이 개수 제한에 밀려 잘려나가지 않았는가 (예: 서버 쏠린 이력에서 풀스택을 증명하는 유일한 경험)
- [ ] 선택된 경험들이 한쪽 결로만 쏠려 있지 않은가 → 쏠렸다면 Step 2.0 인벤토리를 재확인했는가

**어필 체크리스트 (사실 나열 탈출)**
- [ ] 각 경험 서술에 "왜 이 기술/판단을 선택했는가"가 담겨 있는가
- [ ] 도입부가 "저는 ~한 사람입니다"로 시작하는가 (스펙 나열 금지)
- [ ] JD의 특정 문장을 직접 인용하고 증거를 제시했는가
- [ ] **"오면 어떤 가치를 주는가 / 뽑으면 어떤 이점이 생기는가"에 회사 입장(ROI)에서 정면으로 답했는가**
- [ ] **내가 이 회사에 줄 수 있는 가치를 구체적 시나리오로 제안했는가 (즉시 기여 / 역할 정의 / 확장 기여)**
- [ ] **"할 수 있다"는 주장이 과거 실증·수치에 근거하고 있는가**
- [ ] **회사가 시킬 일을 받아 적는 톤이 아니라, 내가 맡을 역할을 정의하는 톤인가**
- [ ] JD 핵심 Required와 충돌하는 공백이 있다면 선제 언급했는가 (없으면 생략 — 사소한 약점은 언급 안 함)

## 출력 — 파일 저장 규칙

생성 결과는 채팅 출력에 그치지 않고 **파일로 저장한다.**

- **경로**: `cover-letters/{회사명}-{포지션}.md` (회사명·포지션은 kebab-case, 한글 회사명은 그대로 허용)
  - 예: `cover-letters/acme-backend-engineer.md`, `cover-letters/스타트업-frontend-developer.md`
  - 사용자가 다른 출력 경로를 지정하면 그 경로를 따른다.
- **파일 구조**: YAML 프런트매터 + 제출용 본문 + 내부 자료(주석 처리)

```markdown
---
title: {회사명} {포지션} 자기소개서
company: {회사명}
position: {포지션}
level: {주니어 / 미드 / 시니어}
created: YYYY-MM-DD
sources: [근거로 사용한 후보자 소스 목록]
---

## 자기소개서 (제출용)

{검증 통과한 본문만}

<!-- ───────── 이하 내부 자료 (제출 제외) ───────── -->
<!--
## JD 분석 (Required/Preferred, Pain Point 가설, 커버 테이블)
## 주장-근거 검증표
## 형식 준수 점검
-->
```

- **제출용 본문**과 **내부 분석·검증표**는 한 파일 안에서 분리한다 (내부 자료는 HTML 주석으로 감싸 제출 시 노출되지 않게).
- 같은 회사·포지션을 다시 생성하면 기존 파일을 덮어쓰기 전에 사용자에게 확인한다.

## 원칙

**하지 않는 것**
- 없는 경험을 있는 것처럼 작성하지 않는다
- 후보자 소스에 없는 수치를 지어내지 않는다 (그럴듯한 추정 금지 — 없으면 정성적 임팩트로 대체)
- "성실하고 열정적인"류의 형용사를 사용하지 않는다
- 자기소개서를 경험의 요약으로 쓰지 않는다
- **핵심 공백(JD 핵심 Required와 직접 충돌)은 숨기지 않는다** — 단, 사소한 약점까지 굳이 먼저 꺼내지는 않는다 (불필요한 약점 강조는 손해)
- Pain Point를 자소서 본문에서 회사의 결함으로 단정하지 않는다 (가설형으로만)

**하는 것**
- 실제 경험을 JD의 Pain Point 언어로 재해석한다
- 기술/판단 선택의 이유와 결과를 함께 서술한다
- 과거 실증을 근거로 "이 회사에서 줄 수 있는 가치와 맡을 역할"을 구체적으로 제안한다
- 면접관이 어차피 발견할 **핵심 공백**은 먼저 꺼내 신뢰를 만든다
