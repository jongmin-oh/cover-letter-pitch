# cover-letter-pitch — 맞춤형 자기소개서/커버레터 작성 스킬

JD(채용공고)와 본인 경험을 입력하면, **서류 합격 확률을 최대화하는 맞춤형 자기소개서**를 생성하는 Claude / Claude Code 스킬입니다.

사실을 나열하는 자소서가 아니라, **후보자의 판단 흐름과 임팩트를 JD의 Pain Point에 맞춰 어필**하는 글을 목표로 합니다. 모든 경험·수치는 사용자가 제공한 소스에서만 가져오며, 없는 내용은 지어내지 않습니다.

## 무엇을 하나

```
JD 입력 + 후보자 경험
  └─► JD 해부 (Required/Preferred/Pain Point 가설)
        └─► 경험 전수 나열 → JD 매핑 → 결정적 증거 선택
              └─► 판단+임팩트 중심 문장 재작성
                    └─► 4단락 본문 작성 (도입 선언 → JD 공명 → 기여 제안 → 약점 선제)
                          └─► 주장-근거 검증 → 검증 통과분만 제출
```

- **JD 해부**: 필수/우대 역량, 시니어리티·컬처 신호, Pain Point 가설을 추출
- **경험 매핑**: 작은 사이드 작업까지 전수 나열한 뒤, 직접성·증거 강도·희소성으로 정렬해 결정적 증거를 보호
- **어필 재작성**: "무엇을 했다"가 아니라 "왜 그 판단을 했고 결과가 무엇인가"로 변환
- **근거 검증**: 본문의 모든 수치·주장을 소스와 대조해, 출처 없는 문장은 제거/교체
- **형식 준수**: 문항형·글자 수 제한을 사전 확인하고 준수

## 설치

### Claude Code
레포를 스킬 폴더로 바로 클론하면 됩니다. 전역으로 쓰려면 `~/.claude/skills/` 아래에, 특정 프로젝트에만 쓰려면 그 프로젝트의 `.claude/skills/` 아래에 둡니다.

```bash
# 전역 설치 (모든 프로젝트에서 사용)
git clone https://github.com/jongmin-oh/cover-letter-pitch ~/.claude/skills/cover-letter-pitch

# 또는 특정 프로젝트에만
git clone https://github.com/jongmin-oh/cover-letter-pitch .claude/skills/cover-letter-pitch
```

설치 후 `/cover-letter-pitch` 로 호출하거나, 그냥 "이 JD로 자기소개서 써줘"라고 요청하면 description 트리거로 자동 호출됩니다.

### Claude (claude.ai)
`SKILL.md`를 커스텀 스킬/프로젝트 지침으로 등록하면 동일하게 동작합니다.

## 사용법

호출 시 다음을 함께 주면 됩니다 (없으면 작성 전에 물어봅니다).

**필수**
- JD 전문 (텍스트 / 파일 경로 / URL)
- 이력서 (파일 경로 또는 붙여넣은 텍스트)
- 목표 언어 (한국어 / 영어)
- 포지션 레벨 (주니어 / 미드 / 시니어 / 리드)

**있으면 더 좋음**
- 포트폴리오 (이력서를 보완하는 자료 — PDF 파일 / 웹 링크 / 이미지 / 텍스트)
- 지원서 문항 (자유 양식 / 문항형)
- 글자 수·분량 제한
- 회사·제품 정보 (회사명, 제품, 최근 방향)
- 이력서 동봉 여부

> 이력서가 불릿만 있어 판단 이유·수치·임팩트가 부족하면, **작성 전에 결정적 경험에 대해 보강 질문을 한 번 모아서** 물어봅니다. 답변은 본문 근거로 쓰이고, 답할 수 없는 수치는 지어내지 않고 정성적 임팩트로 대체합니다.

### 예시 프롬프트

```
아래 JD로 자기소개서를 써줘. 이력서는 resume.pdf, 포트폴리오는 https://my.portfolio 참고.
포지션 레벨: 미드 / 언어: 한국어
지원서 문항: 자유 양식 / 글자 수 제한: 1000자

[JD 전문]
...
```

> 포트폴리오는 선택입니다. 이력서만 줘도 동작하며, 부족한 근거는 작성 전 질문으로 채웁니다.

## 출력

`cover-letters/{회사명}-{포지션}.md` 에 저장됩니다.
파일 안에서 **제출용 본문**과 **내부 분석·검증표**가 분리되며, 내부 자료는 HTML 주석으로 감싸 제출 시 노출되지 않습니다.

## 핵심 원칙

- 없는 경험·수치를 지어내지 않는다 (없으면 정성적 임팩트로 대체)
- 회사의 약점(Pain Point)을 단정하지 않는다 — 가설형으로만, 기여 제안의 도입부로
- 핵심 공백(JD 핵심 Required와 충돌)은 숨기지 않고 먼저 꺼내 신뢰를 만든다
- "성실하고 열정적인"류의 빈 형용사를 쓰지 않는다
